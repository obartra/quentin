# Low-code content editing: Astro + Keystatic

## Why

The site was hand-authored HTML: every headline, bio line, case study, and
testimonial lived inside the markup. Changing a word meant editing HTML. This
migration keeps the exact design and output but moves all the **words and image
paths into editable content files**, with a form-based admin (Keystatic) on top,
so a non-technical editor can maintain the site without touching code.

The design system (`public/assets/css/style.css`), the behavior
(`public/assets/js/*.js`), the SEO layer, and the galleries are
all preserved unchanged. The build reproduces the same flat `*.html` pages, so the
canonical URLs, the sitemap, and both validators keep passing.

## Architecture

```
content/*.yaml            <- the editable content (words + image paths)
keystatic.config.ts       <- schema: what fields exist + the admin UI
src/lib/content.ts        <- build-time reader over content/ (no server)
src/layouts/BaseLayout.astro   <- per-page <head>: SEO, OG, JSON-LD, favicons
src/components/*.astro     <- Header, Footer, Cta
src/pages/*.astro          <- one per page; renders content into the original markup
public/                    <- assets/css, assets/js, assets/img, favicons, manifest,
                              robots.txt, sitemap.xml (copied verbatim into dist/)
        |
        |  npm run build   (Astro, static; Keystatic admin excluded)
        v
dist/*.html + dist/assets  <- deployed to GitHub Pages; validated by tools/*.py
```

Three ideas make this work:

1. **Content is data.** Each page is a Keystatic *singleton* (a YAML file under
   `content/`); repeating things (proof points, cases, testimonials, talk topics,
   galleries, timeline) are *arrays* inside those files. Editing an array in the
   admin adds/removes/reorders items.

2. **Pages render data into the original markup.** The `.astro` pages read content
   at build time via Keystatic's reader API (`@keystatic/core/reader`) and emit the
   same classes and structure the site shipped by hand. Rich fields (a bio
   paragraph, a pull quote with a `<span>`) are stored as small HTML snippets and
   rendered with `set:html`; everything else is plain text.

3. **SEO stays in code.** Titles/descriptions/OG text are editable, but the
   technical layer (canonical URL, the full OG/Twitter set, robots, favicons, and
   the JSON-LD graph keyed to `https://quentinfears.com/#person`) is generated by
   `BaseLayout.astro` and `src/lib/jsonld.ts`. Editors cannot desync it.

## The admin

- **Local mode (default).** `npm run dev`, then open `/keystatic`. Edits write to
  the `content/*.yaml` files; commit and push to publish. No server, no database,
  no account, fully free and open source. This is the current setup.
- **Hosted admin (browser, no dev server).** `npm run build:admin` builds the same
  app with the Keystatic admin as SSR routes behind the Netlify adapter and GitHub
  storage, so a non-technical editor edits from a URL and Save commits to the repo.
  Deploy that to a free Netlify project; the public site stays on GitHub Pages. Full
  step-by-step (GitHub App + env vars) is in
  [hosted-admin.md](hosted-admin.md).

The public site build deliberately **excludes the admin**: `npm run build` sets
`KEYSTATIC_ADMIN=0`, so `astro.config.mjs` skips the Keystatic + React integrations
and the `/keystatic` (SSR) routes, keeping the deployed site a pure, adapter-free
static export. The admin's SSR needs (adapter + serverless function) only apply to the
separate `build:admin` output on Netlify.

## Invariants preserved

- **Flat URLs.** `build.format: 'file'` emits `about.html`, `work.html`, ... so
  canonical/`og:url`, the sitemap, and internal links are unchanged.
- **Relative links.** Every internal link and asset ref stays relative, so the site
  works both under the GitHub Pages subpath (`obartra.github.io/quentin/`) and at
  the apex domain. `base` stays `/`.
- **Gate <-> indexing coupling.** The password gate has been removed and pages ship
  `robots: index, follow`. `tools/seo_check.py` still enforces the coupling, so a
  gated page can never be left indexable (or an ungated page `noindex`).
- **Validators.** `tools/validate_site.py` and `tools/seo_check.py` now take an
  optional target dir and default to `dist/` when present. CI builds, then runs both
  against `dist/`. `deploy.yml` builds and publishes `dist/`.

## Editing recipes

- **Change a headline / bio / testimonial:** `/keystatic` in dev, edit the page,
  save. Or edit the matching `content/*.yaml` by hand.
- **Add a case study / talk topic / timeline entry:** open the page in the admin,
  add an array item.
- **Add a gallery:** edit the `Galleries` singleton — a set is a portfolio category
  (`key`, `title`) containing photo shoots (`slug`, `title`, `images`). Reference the
  category key from a Work case or archive item's "gallery" field; an archive item's
  "shoot" field deep-links to one shoot's image set.
- **Add a whole new page:** add a singleton to `keystatic.config.ts`, a
  `content/<page>.yaml`, and a `src/pages/<page>.astro` that renders it through
  `BaseLayout` (which supplies the SEO/JSON-LD); add the URL to
  `public/sitemap.xml`.
- **Change the domain:** update `SITE_ORIGIN` in `src/lib/content.ts`, `BASE` in
  `tools/seo_check.py`, and `public/robots.txt` + `public/sitemap.xml`.

## Tradeoffs

- A build step and npm dependencies now exist (the previous site had neither). This
  is the deliberate cost of form-based editing; the deployed artifact is still
  static.
- A few rich blocks (the About biography, the TIME pull quote, case detail rows)
  are edited as small HTML snippets rather than fully structured fields. This keeps
  the exact typography while staying editable.
- Output is not byte-identical to the old hand-authored HTML (Astro normalizes some
  entities and whitespace), but it is visually identical and passes both validators.
